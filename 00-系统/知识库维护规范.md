# 知识库维护规范

## 1. 这个仓库应该解决什么问题

这个仓库不是“今天学了什么”的流水账，而是一个长期可复用的算法知识系统。

它应该帮助我回答这些问题：

- 某个算法的核心思想是什么？
- 这个算法适合解决什么类型的问题？
- 时间复杂度和空间复杂度是多少？
- 常见错误在哪里？
- 有没有可复用模板？
- 我以前在哪些题目中用过它？
- 考研机试中会怎么考？

## 2. 笔记的基本要求

每篇正式笔记至少应包含：

- 主题
- 适用场景
- 核心思想
- 复杂度
- 常见坑
- 相关题目或例子
- 相关链接

不要只复制代码。代码只是结果，真正有价值的是：

- 为什么这样写
- 为什么会错
- 怎么判断使用这个方法
- 下次如何快速复现

C++ 语言基础笔记要适合初学者复习，避免写成百科式长文。整理来自 `cpp-Learning` 的案例时，只写源码里真实出现过的内容。

推荐使用这个结构：

- 学习主题
- 核心结论
- 为什么会这样
- 案例对应
- 易错点
- 关键代码摘录
- 复习问题
- 标签

关键代码摘录只保留和知识点直接相关的片段，不整段复制源码，也不要只放跨仓库源码链接。

## 3. 学习闭环

这个知识库优先服务一条轻量、可长期坚持的学习链路：

```text
C++ 基础
        ↓
STL 常用能力
        ↓
数据结构与算法专题
        ↓
LeetCode / 考研机试题目
        ↓
错题复盘
        ↓
可复用模板
```

不同类型的内容不要混写：

- 新知识点写成概念笔记，放入 `C++/`、`STL/`、`数据结构/` 或 `算法/`。
- 刷题过程写成题解笔记，放入 `LeetCode/` 或 `考研机试/`。
- 做错的题写成错题复盘，放入 `错题/`，重点记录错误原因和下次识别信号。
- 多次复用、已经稳定的写法再沉淀到 `模板/`。

一篇错题复盘不等于一篇算法专题；一篇算法专题也不等于代码模板。它们可以互相链接，但各自承担不同任务。

## 4. 文件应该放在哪里

优先按“知识类型”归档：

- 语言知识放 `C++/`
- STL 用法放 `STL/`
- 数据结构概念放 `数据结构/`
- 算法思想放 `算法/`
- 题目记录放 `LeetCode/`
- 错误复盘放 `错题/`
- 可复用代码放 `模板/`
- 考研机试专项放 `考研机试/`

如果当前仓库暂时还没有 `算法/` 目录，首次写算法专题时再创建该目录，不需要为了空目录强行维护结构。

如果一篇内容跨多个主题，选择它最主要的归属，然后用 Obsidian 双链连接其他主题。

目前 C++ 语言基础笔记默认放在 `C++/`，文件名使用中文概念名，例如 `指针基础.md`、`数组和指针.md`、`引用基础.md`。

写新笔记前，先检查当前目录、已有命名和相近主题；如果是根据源码整理笔记，还要先只读查看 `D:\Cloud\OneDrive\文档\Study\Algorithm\cpp-Learning` 中对应案例。

## 5. Obsidian 使用规则

推荐使用三种连接方式：

1. 文件夹：负责大分类。
2. 标签：负责快速筛选。
3. 双链：负责建立知识关系。

示例：

```markdown
相关：

- [[二分查找]]
- [[lower_bound 和 upper_bound]]
- [[有序数组]]

标签：

#算法/二分 #复杂度/O(logn) #题型/查找
```

手动索引使用普通 Markdown 链接或 Obsidian 双链维护，不依赖 Dataview 等插件。推荐把 `00-系统/索引-学习入口.md` 作为日常入口。

## 6. Git 提交规则

每次提交尽量只做一类事情。

好的提交示例：

```text
docs: add binary search note
docs: add vector STL summary
fix: correct dp transition in knapsack note
chore: add note templates
```

不推荐：

```text
update
修改
乱七八糟整理
```

Git 提交信息的意义是：以后回看历史时，能知道当时改了什么。

## 7. 长期维护习惯

建议每周做一次小整理：

- 检查是否有散落文件
- 检查是否有重复主题
- 给新笔记补标签
- 更新 `00-系统/索引-学习入口.md`
- 把错题沉淀到对应算法专题
- 把成熟代码提炼到 `模板/`

长期知识库不是一次建好，而是不断修剪、连接和升级。
