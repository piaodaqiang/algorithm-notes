# 知识库维护规范

## 1. 这个仓库应该解决什么问题

这个仓库不是“今天学了什么”的流水账，而是一个长期可复用的算法知识系统。

它应该帮助我回答这些问题：

- 某个算法的核心思想是什么？
- 这个算法适合解决什么类型的问题？
- 时间复杂度和空间复杂度是多少？
- 常见错误在哪里？
- 有没有可复用模板？
- 我以前在哪些题目中用过它？
- 考研机试中会怎么考？

## 2. 笔记的基本要求

每篇正式笔记至少应包含：

- 主题
- 适用场景
- 核心思想
- 复杂度
- 常见坑
- 相关题目或例子
- 相关链接

不要只复制代码。代码只是结果，真正有价值的是：

- 为什么这样写
- 为什么会错
- 怎么判断使用这个方法
- 下次如何快速复现

## 3. 文件应该放在哪里

优先按“知识类型”归档：

- 语言知识放 `C++/`
- STL 用法放 `STL/`
- 数据结构概念放 `数据结构/`
- 算法思想放 `算法/`
- 题目记录放 `LeetCode/`
- 错误复盘放 `错题/`
- 可复用代码放 `模板/`
- 考研机试专项放 `考研机试/`

如果一篇内容跨多个主题，选择它最主要的归属，然后用 Obsidian 双链连接其他主题。

## 4. Obsidian 使用规则

推荐使用三种连接方式：

1. 文件夹：负责大分类。
2. 标签：负责快速筛选。
3. 双链：负责建立知识关系。

示例：

```markdown
相关：

- [[二分查找]]
- [[lower_bound 和 upper_bound]]
- [[有序数组]]

标签：

#算法/二分 #复杂度/O(logn) #题型/查找
```

## 5. Git 提交规则

每次提交尽量只做一类事情。

好的提交示例：

```text
docs: add binary search note
docs: add vector STL summary
fix: correct dp transition in knapsack note
chore: add note templates
```

不推荐：

```text
update
修改
乱七八糟整理
```

Git 提交信息的意义是：以后回看历史时，能知道当时改了什么。

## 6. 长期维护习惯

建议每周做一次小整理：

- 检查是否有散落文件
- 检查是否有重复主题
- 给新笔记补标签
- 把错题沉淀到对应算法专题
- 把成熟代码提炼到 `模板/`

长期知识库不是一次建好，而是不断修剪、连接和升级。
